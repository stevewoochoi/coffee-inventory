# STORE_ORDERING_ENHANCEMENT.md — 매장 수주 중심 고도화 기획서

> **참고**: 할리스 점주앱 기획안(24p), 현재 Coffee Store 대시보드 스크린샷, 기존 V5.1 코드베이스
> **핵심**: 매장 사용자가 **발주 → 주문현황 → 입고 → 클레임** 사이클을 하나의 흐름으로 체감할 수 있도록 UI/UX 재구성

---

## 1. 현황 분석

### 1-A. 현재 매장(StoreLayout) 네비게이션 — 9개 1단 나열

```
[대시보드] [재고 현황] [입고] [폐기] [발주 현황] [유통기한] [클레임] [실사] [리포트]
```

**문제점**:
- 발주 사이클(발주→주문현황→입고→클레임)이 4개 다른 메뉴에 흩어져 있음
- "발주 현황"을 눌러야 비로소 "신규 발주"로 이동 가능 → 진입 경로가 2단계
- 주문 이후 입고를 확인하려면 별도 "입고" 메뉴 → 연결감 부족
- 재고 관련(재고현황/유통기한/실사)도 흩어져 있음
- 모바일 하단 탭 9개 → 터치 타겟 좁고, 스크롤 필요

### 1-B. 할리스 점주앱 구조 (참고 대상)

할리스는 **하단 탭바 4개 + 전체메뉴(더보기)** 패턴:

```
하단 탭: [홈] [발주하기] [공지사항] [전체메뉴]
```

전체메뉴(더보기)에서 6개 그룹 카드:
- **수주관리**: 주문등록, 주문현황, 입고조회, 클레임등록
- **매출관리**: 매출요약, 영업일보, 기간대비 매출
- **판매관리**: 품절관리, 키오스크 터치키
- **재고관리**: 전수 재고실사 등록, 점간 이동
- **청구관리**: 여신현황, 청구서 조회
- **커뮤니티**: 게시판, 프로모션, AS등록

**핵심 인사이트**:
- "발주하기"가 하단 탭의 **고정 위치** — 매일 쓰는 핵심 기능은 1탭 접근
- 수주관리가 **주문등록 → 주문현황 → 입고조회 → 클레임등록** 순서로 업무 흐름과 동일
- 입고조회에서 날짜/상태 필터 + 출고확정/미확정 배지
- 클레임 유형이 16종으로 세분화 (파손, 유통기한, 표시사항, 발주오류, 이물질 등)
- 메인화면에 매출요약 + 수발주현황 + 여신현황 → 한눈에 파악

## 2. 제안: StoreLayout 네비게이션 재구조화

### 2-A. 모바일 하단 탭바 — 4개로 축소

```
[홈] [발주] [주문내역] [더보기]
```

| 탭 | 라우트 | 설명 |
|---|--------|------|
| 홈 | `/store/dashboard` | 대시보드 (오늘 할일 + 재고현황 + 최근주문) |
| 발주 | `/store/ordering/new` | **바로 신규발주** (현재는 2단계 필요) |
| 주문내역 | `/store/ordering` | 발주현황 + 주문이력 통합 |
| 더보기 | `/store/menu` | 전체메뉴 그리드 (할리스 스타일) |

**변경 근거**:
- 매장 직원이 가장 많이 쓰는 기능은 **발주 생성**과 **주문 확인** → 1탭 접근
- 입고/클레임/재고/실사/리포트는 "더보기"에서 접근 (매일 쓰지는 않음)
- 할리스와 동일 패턴: 핵심 3개 + 전체메뉴 1개

### 2-B. 데스크톱 상단 헤더 — 5개 그룹

```
[대시보드] [수주관리▾] [재고관리▾] [리포트] [더보기]
```

**수주관리 ▾**:
- 신규 발주 `/store/ordering/new`
- 주문현황 `/store/ordering`
- 입고확인 `/store/receiving`
- 클레임 `/store/claims`

**재고관리 ▾**:
- 재고 현황 `/store/inventory`
- 유통기한 `/store/expiry`
- 재고 실사 `/store/physical-count`
- 폐기 `/store/waste`

## 3. 발주(주문등록) 고도화 — 할리스 기획서 반영

### 3-A. 현재 vs 개선

| 영역 | 현재 (V5.1) | 개선 제안 | 할리스 참고 |
|------|-----------|---------|-----------|
| 진입 | 발주현황 → 신규발주 (2단계) | 하단탭 "발주" 1탭 진입 | 하단탭 "발주하기" |
| 납품일 선택 | 달력에서 날짜 선택 | 달력 + **배송가능일 색상 표시** | 달력에 배송가능 표기 |
| 상품 검색 | 텍스트 검색 + 카테고리 탭 | 동일 + **최근주문수량/최대주문수량** 표시 | 연관검색어 + 코드/상품명/규격 |
| 장바구니 | 플로팅 바 | 동일 + **사진보기/리스트보기 토글** | 2가지 뷰 제공 |
| 발주 전송 | 확정 버튼 | 동일 + **전송대기 단계** 추가 | 전송대기 → 발주전송 2단계 |
| 최근 주문 | 재발주 기능 있음 | 동일 | 수발주현황에서 접근 |

### 3-B. 신규: 전송대기(Draft) → 발주전송(Confirm) 분리

할리스에서는 장바구니에 담은 상태가 **전송대기**이고, 명시적으로 **발주 전송**을 해야 본사에 접수됩니다.
현재 시스템의 DRAFT → CONFIRMED 흐름을 사용자에게 더 명확히 보여주는 것입니다.

```
현재: [상품선택] → [장바구니] → [발주확정(CONFIRMED)]
개선: [상품선택] → [장바구니(DRAFT)] → [전송대기 목록] → [발주전송(CONFIRMED)]
```

**전송대기 목록** (OrderingPage에 통합):
```
┌──────────────────────────────────────────────────┐
│ 주문 관리                                        │
├──────────────────────────────────────────────────┤
│ [전송대기 2건] [접수완료] [배송중] [입고완료] [전체] │
│                                                  │
│ ┌── 전송대기 ─────────────────────────────────┐   │
│ │ 납품일: 3/7(금) | 5개 상품 | ₩235,950     │   │
│ │ 마감시간: 3/5 09:00 (D-1)                  │   │
│ │ [수정] [삭제] [🚀 발주 전송]                │   │
│ └────────────────────────────────────────────┘   │
│                                                  │
│ ┌── 접수완료 ─────────────────────────────────┐   │
│ │ #42 | 납품일: 3/6(목) | 8개 상품 | ₩420,000│   │
│ │ 상태: ✅ 접수완료 → 출고준비중              │   │
│ │ [상세보기]                                  │   │

└──────────────────────────────────────────────────┘
```

할리스처럼 **조회/추적** 기능을 강화합니다.

### 4-C. 핵심 개선사항

1. **기간 필터**: 납품일자 범위로 조회 (현재는 발주 건별 입고만)
2. **상태 배지**: 출고확정(녹색) / 미확정(노란색) / 주문(파란색) / 반품(빨간색)
3. **아코디언 상세**: 각 주문건 터치 → 입고정보/출고정보 펼침
4. **입고 확인 빠른 액션**: 미확정 상태에서 바로 입고 처리 버튼
5. **불일치 알림**: 주문수량 ≠ 입고수량 시 빨간 경고 배지

## 5. 클레임 고도화

### 5-A. 현재 vs 할리스 비교

| 영역 | 현재 | 할리스 | 개선 제안 |
|------|------|-------|---------|
| 클레임 유형 | 6종 (DEFECTIVE, WRONG_ITEM, SHORTAGE, DAMAGE, QUALITY, OTHER) | 16종 (반품 7종 + 차감요청 3종 + 일반문의 4종) | **10종으로 확장** |
| 상태 | 4단계 (SUBMITTED→IN_REVIEW→RESOLVED→CLOSED) | 3단계 (접수→처리중→답변완료) | 유지 + **조치예정일** 추가 |
| 상품 선택 | 최근 주문 기반 | **입고 일주일내 상품만** 조회 | 입고 기반으로 변경 |
| 사진 | URL 입력 | **앱 실행 중 바로 촬영 업로드** | 카메라 캡처 추가 |
| 기간 조회 | 상태 필터만 | **월별 조회 + 신청일자 필터** | 기간+상태 복합 필터 |

### 5-B. 클레임 유형 확장

```typescript
// 현재 6종 → 10종으로 확장
const CLAIM_TYPES = [
  // 반품 관련
  'DAMAGE',           // 파손
  'EXPIRY_ISSUE',     // 유통기한(선입선출)
  'LABELING',         // 표시사항
  'ORDER_ERROR',      // 발주오류
  'DEFECTIVE_FOOD',   // 식품 불량품
  'DEFECTIVE_NONFOOD',// 비식 불량품
  'FOREIGN_MATTER',   // 이물질
  // 차감/수량
  'SHORTAGE',         // 미입고/수량부족
  'OVER_DELIVERY',    // 과입고
  // 기타
  'OTHER',            // 기타/일반문의
];
```

## 6. 대시보드 개선

### 6-A. 현재 대시보드 vs 할리스 메인

현재: 오늘 할일 카드 5개 + 재고현황 바 + 7일소비추이 + TOP5 + 빠른액션 5개
할리스: 매출요약 + 공지사항 + 매출현황(6개월) + 여신현황 + 수발주현황

## 7. 구현 범위 정리

### Phase A: 매장 네비게이션 재구조화 (TASK-110)

```
변경 파일:
- StoreLayout.tsx — 하단 탭 4개로 축소 + 데스크톱 그룹 드롭다운
- 신규: /store/menu → StoreMenuPage.tsx (전체메뉴 그리드)
- App.tsx — 라우트 추가 (/store/menu)
- ko.json / ja.json / en.json — 메뉴 키 추가

하단 탭 (모바일):
  [홈 /store/dashboard]
  [발주 /store/ordering/new]
  [주문내역 /store/ordering]
  [더보기 /store/menu]

데스크톱 헤더:
  [대시보드] [수주관리▾] [재고관리▾] [리포트]
```

### Phase B: 발주 흐름 개선 (TASK-111)

```
변경 파일:
- OrderingPage.tsx — 전송대기/접수완료/배송중/입고완료 상태별 탭
- NewOrderPage.tsx — Step 2 카드에 최근주문수량/최대주문수량 표시
- 장바구니에 사진/리스트 뷰 토글 추가
- OrderDetailPage.tsx — 마감 전 수정 UI 강화 (인라인 수량 조정)
- ordering API — getOrderHistory에 상태별 집계 추가

백엔드 변경 없음 (기존 DRAFT/CONFIRMED 상태 활용)
프론트엔드 UX 개선만.
```

### Phase C: 입고조회 통합 (TASK-112)

```
변경 파일:
- ReceivingPage.tsx — 기간 필터 + 상태 배지 + 아코디언 상세 추가
- receiving API 확장:
  GET /api/v1/receiving/history?storeId&from&to&status&keyword
  → 기간별 입고 이력 조회 (기존 pending만 조회 → 이력 전체로 확장)

백엔드 추가:
- ReceivingController에 history 엔드포인트 추가
- DeliveryRepository에 기간/상태 조회 쿼리 추가

응답에 추가 필드:
- orderPlanId, orderDate (발주일)
- deliveryDate (납품일)  
- fulfillmentStatus (출고확정/미확정)
- matchStatus (일치/불일치 — 주문수량 vs 입고수량)
```

### Phase D: 클레임 고도화 (TASK-113)

```
변경 파일:

백엔드:
- ClaimType enum 확장 (6종 → 10종)
- claim 테이블에 estimated_resolve_date 컬럼 추가 (조치예정일)
- GET /api/v1/claims — from/to 기간 필터 파라미터 추가
- claim 신청 시 입고 기반 상품 선택 API:
  GET /api/v1/receiving/recent-items?storeId&days=7
  → 최근 7일 입고 상품 목록

프론트엔드:
- ClaimsPage.tsx — 월별 조회 + 진행상태 프로그레스 표시
- NewClaimPage.tsx
  - Step 1: 입고 기반 상품 선택으로 변경
  - Step 2: 클레임 유형 10종 그리드
  - 사진 촬영: <input type="file" capture="environment" accept="image/*">
- ClaimDetailPage.tsx — 조치예정일 표시 + 진행 타임라인
```

### Phase E: 대시보드 데이터 확장 (TASK-114)

```
변경 파일:

백엔드:
- GET /api/v1/store/dashboard — 응답에 추가:
  recentOrderDate, recentReceivingDate,
  monthlyOrderCount, monthlyOrderAmount,
  nextDeliveryDate, nextDeadline

프론트엔드:
- DashboardPage.tsx — "최근 주문" 섹션 추가
```

## 9. 구현 우선순위

```
Phase A (네비게이션) — 프론트엔드만, 백엔드 변경 없음
  → 가장 먼저. 다른 Phase와 독립적.

Phase B (발주 흐름) — 프론트엔드만
  → Phase A와 동시 가능. 기존 API 그대로 활용.

Phase C (입고조회) — 백엔드 1개 API + 프론트엔드
  → Phase A 이후.

Phase D (클레임) — 백엔드 마이그레이션 + API + 프론트엔드
  → 가장 규모 큼. Phase C 이후.

Phase E (대시보드) — 백엔드 API 확장 + 프론트엔드
  → 마지막. 가장 간단.
```

## 10. 검증 기준

```
1. 모바일 하단 탭 4개 정상 동작 (발주 1탭 진입)
2. 전체메뉴 그리드에서 모든 기능 접근 가능
3. 발주 → 주문현황 → 입고확인 → 클레임 전체 사이클 2분 이내 완료
4. 입고 기간 조회: 1주일 범위 필터링 정상
5. 클레임 10종 유형 선택 + 사진 촬영 업로드 정상
6. 대시보드에 최근 주문/입고 정보 표시
7. 기존 URL 직접 접근 (/store/ordering/new 등) 하위호환 유지
8. ko/ja/en 3개 언어 정상
9. npm run build 성공
```
